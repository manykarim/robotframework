## Context

The MkDocs site has two visual regressions versus `pekkaklarck/manual`:

1. **Section index pages** — `reorganize.py` step 3 (lines 118-128) generates section `index.md` files with only a bare `# Section Name` heading. The `pipeline.py` CLEAN stage (lines 103-126) deletes all `.md` files except `docs/index.md`, so any hand-authored content on section index files is lost every pipeline run.

2. **OCRA font** — `assets/extra.css` declares `font-family: OCRA` for `h1` but does not include a `@font-face` rule or a font file. The font is installed on the developer's machine (OCRAEXT.TTF / OCRAStd.otf) so local `mkdocs serve` shows OCRA headings; CI and GitHub Pages environments have no such font and fall back to the browser default.

## Goals / Non-Goals

**Goals:**
- Section `index.md` files generated by the pipeline contain an intro paragraph and a child-page TOC (consistent with the summaries committed in `fd99bc5e1`)
- OCRA font renders on GitHub Pages (CI environment) without any local font installation
- Both fixes follow the pipeline-first principle: no hand-authored file is patched directly

**Non-Goals:**
- Changing section content beyond the intro + TOC generated by the pipeline
- Matching manual's exact prose style (that belongs in a separate content review)
- Changing how other pages (non-index) are generated

## Decisions

### D1 — Store section intro paragraphs in `reference_map.json`

`reorganize.py` already loads `reference_map.json`. Adding a `"section_intros"` key (section_dir → intro text) keeps all section metadata in one place and makes the generation deterministic. The alternative — a separate config file — creates another file to maintain.

### D2 — Generate child-page TOC from `mkdocs.yml` nav

`reorganize.py` could enumerate the target directory's `.md` files or read `mkdocs.yml`'s `nav` block. Reading `mkdocs.yml` nav is more reliable: it gives titles and ordering without depending on filesystem sort order. The nav is already the authoritative order.

Alternative: hard-code TOC in `reference_map.json`. Rejected — duplicates the nav structure.

### D3 — Always overwrite section index files in `reorganize.py`

Currently step 3 creates only if `not idx.exists()`. We must also regenerate so that updated intros are applied on each pipeline run. The CLEAN stage removes the file first, so `reorganize.py` step 3 will always hit the `not idx.exists()` branch after a fresh pipeline. For `--dry-run` / `--report` consistency, retain the `if not idx.exists()` guard but ensure CLEAN removes the file unconditionally (see D4).

### D4 — Expand `preserved_top_level` to be an empty set for section indexes

Section `index.md` files live at depth 2 (e.g. `creating-test-data/index.md`), not depth 1 — so `preserved_top_level` doesn't protect them. No change is needed to the CLEAN stage for the section index files — they are already deleted. Only the root `docs/index.md` is preserved by the current code, and that is correct.

Conclusion: CLEAN stage needs no change. Only `reorganize.py` step 3 needs to write richer content.

### D5 — Bundle `OCRAStd.otf` in `docs/assets/fonts/`

Copy `OCRAStd.otf` (available at `/home/many/.local/share/fonts/OCRAStd.otf`) to `doc/userguide-mkdocs/docs/assets/fonts/OCRAStd.otf` and add a `@font-face` rule to `assets/extra.css`. This is self-hosted: no CDN dependency, works on CI, works on air-gapped deployments.

Alternative: load OCRA from a CDN (e.g. Google Fonts). OCRA is not on Google Fonts. Alternative: use a web-licensed OCRA variant. Rejected for complexity.

## Risks / Trade-offs

- **Font file size**: OCRAStd.otf is ~38 KB — negligible page-weight impact.
- **nav read coupling**: If `mkdocs.yml` nav structure changes, `reorganize.py` must produce matching output. Low risk — nav changes are rare and intentional.
- **Intro text drift**: Intros stored in `reference_map.json` may become stale if section scope changes. Acceptable: same risk as any static description.
- **OFL/font license**: OCRAStd.otf is under OFL-1.1 (open), web embedding is permitted.
